# thorvg-sys

> **⚠️ Work in progress** — This crate is under active development and not yet ready for production use. APIs may change without notice.

Raw FFI bindings to [ThorVG](https://github.com/thorvg/thorvg), generated by [`bindgen`](https://github.com/rust-lang/rust-bindgen) from the C API (`thorvg_capi.h`).

For a safe, idiomatic Rust API, use the [`thorvg`](https://crates.io/crates/thorvg) crate instead. This crate exists for consumers that need direct access to the C surface or to build alternative wrappers on top of it.

## Versioning

This crate carries its own SemVer; the bundled ThorVG C++ release is
recorded as build metadata. For example, `0.2.0+thorvg-1.0.6` is crate
`0.2.0` bundling ThorVG `1.0.6`. See [`CHANGELOG.md`](CHANGELOG.md).

## Build modes

| Mode | Trigger | What happens |
|------|---------|--------------|
| Vendored (default) | `vendored` feature | ThorVG is compiled from the vendored submodule via [`cc`](https://crates.io/crates/cc) |
| System | `--no-default-features` | Links a system-installed ThorVG via `pkg-config` (>= 1.0) |

Vendored builds need a C++ cross compiler plus `libclang` (used by
[`bindgen`](https://rust-lang.github.io/rust-bindgen/requirements.html)
at build time to parse `thorvg_capi.h`).  No meson, ninja, or
pkg-config required.

## Features

| Feature | Default | Description |
|---------|---------|-------------|
| `vendored` | ✅ | Build ThorVG from source via the `cc` crate |
| `lottie` | ✅ | Lottie animation loader |
| `svg` | ✅ | SVG loader |
| `png` | ✅ | Built-in PNG loader (lodepng, no system dep) |
| `fonts` | ✅ | SFNT / OTF / TTF font loaders |
| `expressions` | ✅ | Lottie expressions via JerryScript (~200 KB RAM) |
| `threads` | ✅ | Multi-threaded TaskScheduler (`std::thread`) |
| `file-io` | ✅ | Filesystem I/O APIs in ThorVG |

Each loader / capability flag toggles the corresponding source files and `#define` in the generated `config.h`. Disable defaults and pick what you need for embedded use:

```toml
[dependencies]
thorvg-sys = { version = "0.2", default-features = false, features = ["vendored", "lottie", "svg"] }
```

## Cross-compilation

Cross-compiler selection follows cc-rs's standard `CC_<triple>` / `CXX_<triple>` contract; no vendor binary names are hardcoded. Put the cross compiler in `.cargo/config.toml`:

```toml
[env]
CC_riscv32imac-unknown-none-elf  = "riscv32-esp-elf-gcc"
CXX_riscv32imac-unknown-none-elf = "riscv32-esp-elf-g++"
```

## `no_std` Support

This crate is `no_std` compatible — the generated bindings use
`core::ffi` rather than `std::os::raw`, and no `std::*` items leak
into the public surface.

For `target_os == "none"` targets, `thorvg-sys` vendors and compiles
[picolibc](https://github.com/picolibc/picolibc) as the libc; the
cross toolchain's `libstdc++` / `libgcc` / `libm` are still used.
See [`docs/bare_metal.md`](docs/bare_metal.md) for the build
pipeline, the runtime stubs we ship, and how to override them from
the consumer crate.

## Building

The vendored ThorVG source and picolibc tree live as git submodules:

```bash
git clone --recurse-submodules https://github.com/goyox86/thorvg-rs
cd thorvg-rs
cargo build -p thorvg-sys
```

`build.rs` regenerates `bindings.rs` and recompiles the C++ archive
on changes to either submodule.
